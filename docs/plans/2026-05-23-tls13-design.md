# TLS 1.3 完整支持设计

**目标：** 在现有内核 HTTP/HTTPS 客户端中增加完整 TLS 1.3 客户端能力，覆盖 full handshake、HelloRetryRequest、PSK/resumption、0-RTT、post-handshake ticket、证书校验、ALPN、HTTP/1.1 与 HTTP/2 收发。

**架构：** TLS 1.3 与 TLS 1.2 共用 WSK、CNG、证书和客户端上层接口，但握手、key schedule、record protection 单独实现。`TlsConnection` 负责协议选择和 I/O 编排，TLS 1.3 细节收敛到 `TlsHandshake13`、`TlsContext` 的 TLS 1.3 secret 状态，以及 `TlsRecordLayer` 的 TLS 1.3 AEAD API。

**技术栈：** Windows kernel C++ `/kernel` 子集，WSK，内核 CNG/BCrypt，AES-GCM，HKDF/HMAC-SHA256/SHA384，ECDHE P-256/P-384/P-521，TLS 1.3 RFC 8446，测试向量参考 RFC 8448。

---

## 范围

- 支持 TLS 1.3 `TLS_AES_128_GCM_SHA256` 和 `TLS_AES_256_GCM_SHA384`。
- ClientHello 发送 `supported_versions`、`key_share`、`supported_groups`、`signature_algorithms`、`psk_key_exchange_modes`、SNI、ALPN、可选 `pre_shared_key` 与 `early_data`。
- 支持服务端协商 TLS 1.3 或回退到现有 TLS 1.2 路径；不把回退当作 TLS 1.3 正常架构的一部分。
- 支持 HelloRetryRequest：按服务端选择的 named group 重新生成 key share，并正确处理 transcript synthetic message hash。
- 支持 full handshake：ServerHello、EncryptedExtensions、CertificateRequest、Certificate、CertificateVerify、Finished。
- 支持 resumption：接收并保存 NewSessionTicket 元数据，后续连接可用 PSK binder 发起恢复。
- 支持 0-RTT：当已有 ticket 允许 early data 且调用方显式启用时，使用 early traffic secret 发送 early application data；服务端拒绝 early data 时停止继续发送并进入普通握手。
- 支持 post-handshake NewSessionTicket；拒绝不支持的 post-handshake CertificateRequest。
- 保持现有证书校验路径，TLS 1.3 Certificate 解析后仍调用 `CertificateValidator`。
- 示例补充 TLS 1.3 HTTPS 与 HTTP/2 ALPN 请求，包括 no-verify 和本地 TLS 1.3 smoke。

## 非目标

- 不新增 WinHTTP、WinINet、SChannel 依赖。
- 不支持 TLS 1.3 客户端证书私钥签名；收到强制客户端证书请求时按协议失败。
- 不支持 ChaCha20-Poly1305，除非后续新增内核可用实现。
- 不引入“临时兜底”或吞错路径；协议不支持时明确返回错误。

## 模块设计

### `crypto::CngProvider`

新增 HKDF 所需的 HMAC 组合能力，不直接暴露可变内部状态：

- `HkdfExtract(hash, salt, ikm) -> prk`
- `HkdfExpand(hash, prk, info, output)`
- 保持 user-mode test stub 可确定输出，方便单元测试覆盖调用路径。

### `tls::TlsContext`

扩展协议、cipher suite 和 TLS 1.3 secret 状态：

- `TlsProtocol::Tls12`、`TlsProtocol::Tls13`
- `TlsCipherSuite::TlsAes128GcmSha256`、`TlsCipherSuite::TlsAes256GcmSha384`
- 保存 client/server random、ECDHE shared secret、early secret、handshake secret、master secret、client/server handshake traffic secret、application traffic secret、exporter secret、resumption master secret。
- 保存 TLS 1.3 ticket：identity、ticket nonce、ticket age add、lifetime、max early data、cipher suite、resumption secret。

TLS 1.2 的 `MasterSecret` 和 key block API 保持兼容，TLS 1.3 使用独立 API 派生 traffic keys。

### `tls::TlsRecordLayer`

保留 TLS 1.2 `ProtectAesGcm` / `UnprotectAesGcm`，新增 TLS 1.3 记录 API：

- `ProtectAesGcm13(plaintext, innerType, writeState, destination)`
- `UnprotectAesGcm13(encrypted, readState, plaintext, output)`

TLS 1.3 记录外层始终使用 `application_data` 和 legacy version `0x0303`。AAD 为外层 5 字节 record header，nonce 为 static IV XOR 64-bit sequence number，解密后去除 padding 并恢复内层 content type。

### `tls::TlsHandshake13`

新增 TLS 1.3 专用握手模块：

- 编码 ClientHello，支持普通、PSK、0-RTT、HelloRetryRequest 第二次 ClientHello。
- 解析 ServerHello 并区分 TLS 1.3 ServerHello、HelloRetryRequest、TLS 1.2 ServerHello。
- 解析 EncryptedExtensions、CertificateRequest、Certificate、CertificateVerify、Finished、NewSessionTicket。
- 提供 HKDF label 派生、Finished verify data、CertificateVerify 签名输入构造。
- 提供 transcript hash helper，含 HelloRetryRequest 的 synthetic message hash。

### `tls::TlsConnection`

`Connect` 默认进入 TLS 1.3 协商：

1. 生成首个 key share 并发送兼容 ClientHello。
2. 读取 ServerHello。
3. 如果服务端选择 TLS 1.2，转入现有 TLS 1.2 编排。
4. 如果收到 HelloRetryRequest，更新 transcript 并发送第二个 ClientHello。
5. 派生 handshake traffic secret，切换到加密握手 record。
6. 读取 EncryptedExtensions、可选 CertificateRequest、Certificate、CertificateVerify、Finished。
7. 验证证书和 CertificateVerify，再验证 Finished。
8. 派生 application traffic secret，发送 client Finished。
9. 如果 early data 被接受，切换 early/application 状态；否则正常发送应用数据。
10. 建立后继续消费 NewSessionTicket 并更新 session cache。

读取应用数据时支持 TLS 1.3 post-handshake handshake record，只处理 NewSessionTicket，其余不支持消息返回明确错误。

### 客户端选项与示例

`TlsClientConnectionOptions` 新增：

- `MinimumVersion`
- `MaximumVersion`
- `EnableSessionResumption`
- `EnableEarlyData`
- `SessionCache`

`HttpsRequestOptions` 透传 TLS 版本和 early data 选项。示例新增：

- TLS 1.3 HTTPS GET
- TLS 1.3 HTTP/2 GET
- TLS 1.3 no-verify GET
- 本地 TLS 1.3 smoke

## 错误处理

- 协议格式错误返回 `STATUS_INVALID_NETWORK_RESPONSE`。
- 不支持的算法、扩展组合或 post-handshake 请求返回 `STATUS_NOT_SUPPORTED`。
- 验签或 Finished 校验失败返回 `STATUS_INVALID_SIGNATURE` 或 `STATUS_INVALID_NETWORK_RESPONSE`。
- 0-RTT 被服务端拒绝不是错误，但调用方必须知道 early data 未被确认；连接继续普通 TLS 1.3。
